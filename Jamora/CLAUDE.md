## Current State

**Phase 1 (storefront MVP) is in progress.** The customer-facing storefront exists in [jamora-web/](jamora-web/) — a Next.js 16 + React 19 + Tailwind v4 app. The backend/CMS, real payments, and a database are **not built yet** (see [plan.md](plan.md) for the full roadmap). Product data is currently mock data in [jamora-web/src/lib/products.ts](jamora-web/src/lib/products.ts), not a database.

### Commands (run inside `jamora-web/`)

```bash
npm run dev     # dev server (Turbopack) at http://localhost:3000
npm run build   # production build — also runs the TypeScript type-check
npm run lint    # ESLint (eslint-config-next); no `next lint`, just `eslint`
npm start       # serve the production build (run `npm run build` first)
```

There is no test runner wired up yet. Verify changes with `npm run build` (catches type errors) and by driving the pages.

### ⚠️ Next.js 16 — not the Next.js in your training data

[jamora-web/AGENTS.md](jamora-web/AGENTS.md) warns that this version has breaking changes. Bundled docs live in `jamora-web/node_modules/next/dist/docs/` — read the relevant guide before writing routing/data code. Landmines already hit:

- **`params` and `searchParams` are `Promise`s** in pages — you must `await` them. Use the global `PageProps<'/route'>` / `LayoutProps<'/route'>` helper types (no import needed; generated by `next dev`/`next build`).
- **Tailwind v4** is configured in CSS via `@theme` in [jamora-web/src/app/globals.css](jamora-web/src/app/globals.css) — there is **no `tailwind.config.js`**. Design tokens (colors, fonts) are CSS variables there; add new tokens to `@theme`, not a JS config.
- React 19's `react-hooks/set-state-in-effect` lint rule fires on the SSR-safe localStorage-hydration pattern (cart, cookie consent). Those specific spots carry justified `eslint-disable` comments — follow that precedent rather than fighting the pattern.

## What Jamora Is

Jamora is a premium e-commerce storefront for an Indonesian herbal/jamu brand targeting the **European market**. Beyond checkout, it is a storytelling platform pairing Indonesian herbal heritage with EU health, aesthetic, and regulatory standards. Brand motto: *Energi, Digestie, Echlibru* (Energy, Digestion, Balance) — these three are also the product benefit categories used across catalog and merchandising.

Full requirements and phasing live in [plan.md](plan.md). Note that plan.md is written in mixed Indonesian/English; the **customer-facing site itself is English-first** (with planned French, Dutch, German localization).

## Deployment

The full stack is containerised for a Linux server via [docker-compose.yml](docker-compose.yml)
(Postgres + Redis + Medusa + storefront). Steps and env in [DEPLOYMENT.md](DEPLOYMENT.md).
Key detail: the storefront reads `MEDUSA_PUBLISHABLE_KEY` **server-side at runtime**
(not `NEXT_PUBLIC_`, so it isn't baked into the image); the Medusa container seeds and
creates the admin user on first boot via its `docker-entrypoint.sh`.

## Architecture

The plan calls for a **headless / decoupled** architecture:

- **Storefront:** Next.js (React) + TailwindCSS — SSR/SSG for load speed and international SEO. **Built** — see [jamora-web/](jamora-web/).
- **Backend & CMS:** Laravel + Filament, or Strapi / MedusaJS — admin dashboard, inventory, educational content. **Not built yet.**
- **Database:** PostgreSQL or MySQL. **Not built yet.**

### Storefront layout (`jamora-web/src/`)

- `lib/products.ts` — the product domain model (`Product`, `Category`, `Certification`), the mock catalog, and `formatEUR`. This is the single source of truth for products until a backend/CMS exists; when the backend lands, keep the types and swap the data source.
- `app/` — App Router routes: `/` (home), `/shop` (catalog, category-filtered via `?c=`), `/product/[slug]` (SSG detail), `/cart`, `/about`.
- `components/` — `cart-context.tsx` is a client-side cart (React context + `useReducer`, persisted to `localStorage`); most other files are presentational. `"use client"` is used only where interactivity requires it — keep pages as Server Components by default.
- The `Energy / Digestion / Balance` taxonomy is centralised in `CATEGORY_META` (labels, the Indonesian pillar names, and per-category accent colors) — reuse it rather than re-deriving category display anywhere.

## Requirements That Shape the Code

These are the non-obvious, cross-cutting constraints from the plan that will affect many parts of the implementation:

- **EU regulatory compliance is a first-class requirement, not a feature toggle.** GDPR cookie consent must be *granular* (per-category opt-in, not a single "OK" button). Prefer privacy-friendly, cookieless analytics (Plausible or Matomo) over aggressive tracking.
- **Multi-currency defaults to EUR (€)** and invoicing must be VAT-friendly / EU-standard.
- **Payments target European methods**, not just cards: via Stripe or Adyen, support Apple/Google Pay plus local rails — iDEAL (NL), Sofort/Klarna (DE/AT), Bancontact (BE).
- **Product data carries compliance metadata:** ingredient transparency, allergens, consumption instructions, and certification badges (Organic, Vegan, EU Compliant, GMP). Model these as structured fields, not free text.
- **Inventory is potentially multi-location** (Indonesia + Europe warehouses) with low-stock warnings and stock-change logging.
- **Product benefit categorization** (Energy / Digestion / Balance) is a core taxonomy used in both catalog navigation and analytics.
